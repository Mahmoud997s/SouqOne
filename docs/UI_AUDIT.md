## 🛑 1. Critical Issues & Duplicates

### 1.1 The "Card" Chaos (Similar UI Repeated)
We have an explosion of card components across the application, many doing almost the exact same thing but tightly coupled to specific features.
* `components/generic-listing-card.tsx`
* `features/listings/components/ListingCard.tsx`
* `features/listings/components/UnifiedCard.tsx`
* `features/rentals/components/rental-card.tsx`
* `features/sale/components/SaleCard.tsx`

**Refactor Suggestion:** Create a single, highly polymorphic `BaseCard` in `/components/ui/card.tsx` that accepts slots (image-slot, badge-slot, price-slot, details-slot). Feature cards should just be wrappers passing specific data to `BaseCard`.

### 1.2 The "Page Shell" Duplication
We have 4 different Page Shells for rendering lists of items:
* `components/listing-page-shell.tsx`
* `features/listings/components/ListingsPageShell.tsx`
* `features/rental/components/RentalPageShell.tsx`
* `features/sale/components/SalePageShell.tsx`

**Refactor Suggestion:** Consolidate into a single `DataViewShell` that accepts generic props for filtering, sorting, and grid rendering.

### 1.3 Exact Name Collisions (Naming Inconsistency)
* `components/empty-state.tsx` vs `features/listings/components/EmptyState.tsx`
* `components/ui/search-bar.tsx` vs `components/layout/navbar/search-bar.tsx` vs `features/listings/components/SearchBar.tsx` vs `features/home/components/search-box.tsx`

**Refactor Suggestion:** Standardize naming. Feature-specific components should be prefixed with their domain (e.g., `HomeSearchBox`, `NavbarSearchBar`) or kept strictly isolated. The generic `EmptyState` should be the only one used; delete the feature-specific one if it doesn't add unique value.
